# Session 36 — Fix WhatsApp Search + Non-English Search Fallback

**Date:** 2026-02-25
**Branch:** `claude/improve-issue-search` (PR #82, merged), `claude/non-english-search-fallback` (PR #83, merged)

## What Was Done

### 1. Word-Based Issue Search (PR #82)

**Problem:** WhatsApp bot users searching for "train cancellation" or "cancelled train" weren't finding the issue "Train Cancellations". The bot told users no match existed.

**Root cause:** Two-part problem:

- SQL LIKE full-phrase matching (`LIKE '%my train keeps getting cancelled%'`) fails for natural language queries
- Bot AI (Claude Sonnet) had no guidance on query construction — it was sending full sentences as search terms

**Fix:**

- `issues.ts`: Word-based search — split query into words, filter stop words, cap at 5 words. AND matching first (all words must match), OR fallback when AND returns nothing with 2+ words
- `issues.ts`: `escapeLike()` function to escape `%`, `_`, `\` in LIKE patterns (security fix)
- `issues.ts`: `parseSearchWords()` for reusable word extraction
- `api/issues/route.ts`: Added `trimAndLimit(search, 500)` input sanitization
- `seed.ts`: Added synonyms for 14 issues that previously had none
- `scripts/seed-synonyms.ts`: New production synonym seeding script (209 synonyms across 50 issues)
- SKILL.md: Updated bot instructions with keyword extraction examples and retry strategy
- 22 new tests for word-based search, stop words, escaping, edge cases
- 3 new bot API search tests

### 2. Non-English Search Fallback (PR #83)

**Problem:** Non-English users searching in their own language (e.g., "Odwołania pociągów" in Polish) got zero results because search only matched English names/synonyms. The translations table had translated names but was only used for display, not search.

**Fix:**

- `issues.ts`: `buildTranslatedLikeClause()` — when `languageCode` is provided and not 'en', adds a subquery against the `translations` table alongside name/synonym matching
- `organisations.ts`: Added full search support (previously category-only filter) with the same word-based AND→OR pattern, plus translation-aware matching
- Bot API: `search_issues` now passes `language_code` through to `getAllIssues()` for server-side translation search
- Web API routes: Both issues and organisations routes accept `locale` query param
- Issues page: Passes `locale` to `getAllIssues()` for server-side search
- Organisations page: Added SearchBar component + search + locale passthrough
- 28 new tests: 12 issue translation search + 16 org search/translation
- Test data: Added Polish/Spanish translations + language reference rows
- Messages: Added `searchPlaceholder` to Organisations section in all 45 locale files

## Decisions Made

1. **Word-based search over full-text search** — SQLite FTS5 would be more powerful but adds complexity. Word-based LIKE with AND→OR fallback handles the actual use case (natural language queries from bot users) well enough.

2. **Translation search is opt-in via languageCode param** — English searches (the majority) don't incur the extra subquery cost. Non-English searches add a simple indexed lookup against the translations table.

3. **Organisations now have search** — The organisations page was the only browsing page without a search bar. Added it here since we were already implementing translation search.

4. **Stop words are English-only** — The stop word list only filters English words. For non-English locales, all words pass through to the query. This is fine because non-English queries hit the translations table where words are meaningful.

5. **Security: parameterised language_code** — Even though language_code comes from server-controlled locale routing (not user input), we parameterise it in SQL queries for defense in depth.
