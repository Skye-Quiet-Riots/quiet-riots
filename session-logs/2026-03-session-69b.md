## Decisions Made

1. **Follow is independent of join:** Following is lighter-weight engagement — users get updates without appearing in the rioter count. 100-follow soft cap to prevent abuse.
2. **Auto-follow on join:** When a user joins an issue, they automatically follow it (auto_followed=1). When they leave, only the auto-follow is removed; manual follows survive.
3. **Flat Zod schemas for bot actions:** Added `follow_issue`, `unfollow_issue`, `get_followed_issues` schemas to `actionSchemas` object — without these, the bot returns 400 validation errors. This was a bug caught during testing.
4. **Issue IDs in tests must match seed data:** Tests initially used `issue-bank` which doesn't exist in `seed-test-data.ts`. Fixed to use `issue-broadband` (valid seed issue). Foreign key constraints catch this — a good safety net.
5. **Follow cap test needs FK bypass:** The 100-follow cap test inserts fake issue IDs into user_follows, which violates FK constraints. Fixed by temporarily disabling `PRAGMA foreign_keys` for that test.

## Surprising Findings

1. **Seed test data has no `issue-bank`** — only `issue-rail`, `issue-broadband`, `issue-flights`, `issue-mobile-data`. Tests written with non-existent IDs fail with `not_found` (not FK errors) because the query checks issue existence first.
2. **Bot action schemas are separate from case handlers** — forgetting to add Zod schemas causes silent 400 "validation failed" errors. The action dispatching code runs fine, but never gets reached.
3. **apply-ui-translations.js expects flat format** — `"Follow.follow": "value"` not nested `{ "Follow": { "follow": "value" } }`. The translation agent produces nested format, requiring a flatten step.
