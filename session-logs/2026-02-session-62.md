# Session 62 — i18n Translation Pipeline Hardening

**Date:** 2026-02-28
**Branch:** `claude/i18n-pipeline-hardening`
**PR:** #154 (merged)

## What Was Done

Implemented a 4-phase defense-in-depth hardening of the i18n translation pipeline, preventing the class of bugs that caused non-English users to see English text on issue pages (PRs #152 and #153).

### Phase 1: CI Key Parity Tests
- Added tests for all 9 keyed sections in `translations/*.json` (was only testing 5)
- New coverage: `actions`, `expert_profiles`, `riot_reels`, `action_initiatives`, `issue_per_riot`
- Added catch-all generic section test that auto-covers any future section
- Tests immediately caught real data issues: 4 locales missing LIKE pattern keys, 19 locales with missing `human_focus` fields (77 total)
- Fixed all 23 affected translation files

### Phase 2: Strict Mode for `--apply`
- Replaced all 9 bare `skipped++` counters with structured `SkipEntry` tracking (locale, section, key, reason)
- Added per-section skip summary report at end of `--apply`
- Added `--strict` flag (exit 1 on skips) — defaults OFF since production has entities not in seed data
- Added `--verbose` flag (print each skip as it happens)

### Phase 3: Key Normalization in translate.ts
- Added `normalizeTranslatedKeys()` to `translate-validation.ts`
- Auto-remaps translated `issue_per_riot` keys back to English via reverse lookup from `issues` section
- Prevents wrong keys from being written to translation files in the first place
- 7 test cases covering: correct keys, reverse lookup, unknown keys, missing keys, LIKE patterns, non-issue sections, mixed scenarios
- Deprecated standalone `normalize-per-riot-keys.js`
- Integrated into translate.ts pipeline (runs after AI translation + validation, before writing)

### Phase 4: Post-Apply DB Coverage Verification
- After `--apply`, prints coverage report: translation counts per entity type across locales
- Flags uneven coverage across locales (min/max mismatch)
- Added `--verify` flag for standalone DB verification without applying
- In `--strict` mode, exits with code 1 if coverage gaps found

## Decisions

1. **`--strict` defaults OFF** — Production has entities (user-suggested issues) not in `perRiotCopy`, so some skips are legitimate. The skip report gives visibility; `--strict` is opt-in for zero-tolerance.
2. **Key normalization only auto-fixes `issue_per_riot`** — Other sections (expert_profiles, riot_reels, etc.) use stable keys (video IDs, expert names) that should never change. Wrong keys in those sections get a warning but no auto-fix.
3. **LIKE pattern keys treated as literal strings** — `%Bus%Cuts` and `%Sewage in Rivers%` are SQL LIKE patterns but stored as literal JSON keys. Tests validate them as-is.
