## Key decisions

| Decision         | Choice                                          | Rationale                                                       |
| ---------------- | ----------------------------------------------- | --------------------------------------------------------------- |
| Password hashing | bcryptjs 12 rounds + SHA-256 pre-hash           | Pure JS for serverless, pre-hash handles >72 byte passwords     |
| Password policy  | Min 10 chars + HIBP breach check                | NIST SP 800-63B compliant, no composition rules                 |
| Rate limiting    | DB-backed for all password routes               | In-memory resets on Vercel cold starts                          |
| Session security | Legacy cookie validates session_version from DB | Fixes gap where password reset didn't invalidate sessions       |
| Phone change     | Step-up auth (5min login recency check)         | Prevents takeover from stale sessions                           |
| Combined input   | `@` always → email; `+digits` → phone           | Simpler than tabs, handles edge cases like `user+tag@gmail.com` |
| Seed data phone  | `user-with-password` uses `+447700900077`       | Avoids collision with `+447700900099` used in phone auth tests  |

## Surprises / Gotchas

- **Phone number collision in seed data:** Initially added `user-with-password` with `+447700900099` — this broke 8 tests because phone auth and phone management tests use that number as an "unregistered phone." Changed to `+447700900077`.
- **Verification token type constraint:** Bot `link_email` action initially used `type='email_link'` which violated the CHECK constraint `IN ('magic_link', 'password_reset', 'email_verify')`. Fixed to use `'email_verify'`.
- **Parallel translation agents:** Used 8 parallel sub-agents to translate 65 keys into 44 locales simultaneously. All completed successfully with validated key parity.
