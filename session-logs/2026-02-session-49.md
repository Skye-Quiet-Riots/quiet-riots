## What Was Done

### 1. WhatsApp Message Delivery Queue (PR #113)

**Problem:** `sendWhatsAppMessage()` calls `/opt/homebrew/bin/openclaw` which only exists on the local Mac. All notification paths from Vercel serverless functions silently failed. This was the same root cause as the OTP delivery bug (session 46).

**Solution:** DB queue polling pattern — same as OTP delivery:

- Migration 023: 4 new columns on `messages` table + partial index
- Query functions: `getUndeliveredMessages()`, `markMessageDelivered()`, `incrementDeliveryAttempt()`, `cleanExpiredDeliveryMessages()`
- Bot API actions: `get_undelivered_messages`, `mark_message_delivered`
- Refactored 4 `notifySetupGuides/notifyUser/notifyShareGuides/notifyShareGuidesQuestion` functions to use DB queue instead of direct `sendWhatsAppMessage()`
- `scripts/deliver-messages.sh` polling script (5s interval via LaunchAgent)
- 27 new tests (unit + integration)

**Design decisions:**

- No `whatsapp_phone` column — JOIN to `users.phone` (DRY, avoids stale data)
- 4-hour delivery window with max 5 retry attempts
- NULL `whatsapp_message` after delivery (defence in depth)
- Separate script from OTP (different intervals, urgency, failure modes)

### 2. Phone Login Session Bug Fix (PR #114)

**Problem:** User (Simon Darling) logged in with phone, received verification code, entered it, was redirected to homepage but appeared NOT logged in.

**Root cause:** Phone/password signin routes only set the legacy `qr_user_id` cookie via `setSession()`, but the client-side UI (nav-bar, auth-gate) uses `useSession()` from `next-auth/react` which reads the Auth.js JWT cookie. Without the JWT cookie, users appeared unauthenticated to all client components.

**Fix:** Updated `setSession()` to encode and set BOTH the legacy cookie AND the Auth.js JWT session cookie using `encode()` from `next-auth/jwt`. Updated all 4 callers: phone signin, password signin, password signup, user create. Also updated `clearSession()` to clear both cookies.

**E2E verified on production:** Sent OTP to +447974766838, entered code in browser, confirmed homepage shows authenticated state (avatar, inbox badge).

## Decisions Made

| Decision              | Choice                                                                | Why                                                            |
| --------------------- | --------------------------------------------------------------------- | -------------------------------------------------------------- |
| JWT encoding approach | `encode()` from `next-auth/jwt`                                       | Uses same encryption as Auth.js — no custom JWT logic needed   |
| Cookie name           | `__Secure-authjs.session-token` (prod) / `authjs.session-token` (dev) | Matches what Auth.js `SessionProvider` reads                   |
| Graceful degradation  | `try/catch` around JWT encoding                                       | If `AUTH_SECRET` missing (test env), legacy cookie still works |
| User info in JWT      | name, email, picture passed to `setSession()`                         | Nav-bar uses `session.user.name` for avatar initial            |

## Gotchas Discovered

- **Dual cookie requirement:** Any code path that logs a user in MUST set both the legacy `qr_user_id` cookie and the Auth.js JWT cookie. The `setSession()` function now handles both automatically.
- **Password login had the same bug:** Not just phone — password signin also only set the legacy cookie. Fixed in the same PR.
