### 2. Localised suggestion review WhatsApp notifications (PR #157)

User noticed that when a Banglish user's suggestion was approved, the WhatsApp notification arrived in English. Fixed by localising all suggestion review notifications.

**Changes:**
- Added 9 new BotMessages keys to `messages/en.json`: `suggestionApproved`, `suggestionRejected`, `suggestionRejectedWithLink`, `suggestionMerged`, `suggestionMoreInfo`, `suggestionGoLive`, `rejectionCloseToExisting`, `rejectionAboutPeople`, `rejectionIllegalSubject`
- Added `whatsappOverride` parameter to `notifyUser()` in bot route ‚Äî allows passing a pre-built localised WhatsApp message instead of deriving from English subject+body
- Updated web review route (`/api/suggestions/[id]/review/route.ts`) to use `getBotMessage(locale, key, params)` for all 4 decision types
- Updated bot route `review_suggestion` handler similarly
- Updated go-live notifications in both web and bot routes
- Added regression test confirming localised approval notification
- Translated all 9 keys to all 55 non-English locales

**Design decisions:**
- Inbox subject/body stays English (separate concern ‚Äî localising inbox is a bigger change)
- Freeform guide text (rejection_detail, more_info notes) stays in English ‚Äî we localise the chrome around it
- Guide and admin notifications stay English (internal staff)

### 3. Translation locale guard (PR #158)

Added `scripts/apply-ui-translations.js` ‚Äî a reusable helper that prevents translation agents from using wrong locale codes.

**Problem:** Translation agents sometimes use wrong codes (`zh` instead of `zh-CN`, `fil` instead of `tl`) or invent locales that don't exist in the project.

**Solution:**
- Script derives the canonical locale list from actual `messages/*.json` files
- `--list-locales` prints the exact list for embedding in agent prompts
- `--check` validates a translation file against expected locales
- Default mode applies translations with validation and JSON integrity verification
- Updated CLAUDE.md UI Translation Protocol to use this script

## Decisions & rationale

1. **`whatsappOverride` vs refactoring `notifyUser`:** Chose a minimal parameter addition rather than restructuring the whole notification system. The bot route's `notifyUser` was designed around English subject+body ‚Üí WhatsApp message. Adding an override parameter keeps backwards compatibility while enabling localisation.

2. **One Task agent for translations:** The UI Translation Protocol says to use one agent, not six. This worked perfectly ‚Äî all 55 locales in one shot.

3. **Script-based locale validation over test:** A script is more useful than a test because it can be run proactively (before applying translations) and provides the locale list for prompts. A test would only catch problems after the fact.
