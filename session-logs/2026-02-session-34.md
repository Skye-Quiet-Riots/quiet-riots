# Session 34 — Fix WhatsApp Bot Returning Issues in English

**Date:** 2026-02-25
**Branch:** `claude/bot-translated-issues`
**PR:** #77 (merged)

## What Was Done

### Bug: WhatsApp bot showed issues in English regardless of user language

When a Polish user asked the bot about issues, the bot would respond in Polish but the issue names, org names, and campaign titles came from the database in English (e.g. "Rail Cancellations" instead of "Odwołania pociągów").

**Root cause:** The bot API tracked users' `language_code` and had translation infrastructure (translations table, `translateEntities()` overlay functions, seed-translations script) — but the data-fetching actions never actually used any of it. The `search_issues`, `get_trending`, `get_issue`, etc. all returned raw English data.

### Fix

1. **Bot API route** (`src/app/api/bot/route.ts`):
   - Added optional `language_code` param to all data-fetching action schemas: `search_issues`, `get_trending`, `get_issue`, `get_actions`, `get_community`, `get_org_pivot`, `get_orgs`, `get_campaigns`
   - Applied `translateEntities()`, `translateEntity()`, `translateCampaigns()`, `translateIssuePivotRows()`, `translateOrgPivotRows()` overlays before returning data
   - `identify` now translates the user's joined issue names using their stored `language_code`

2. **Translation overlay** (`src/lib/queries/translate.ts`):
   - Made `translateOrgPivotRows()` generic — now works with any `{ issue_id, issue_name }` shape (previously required `OrgPivotRow` type)
   - This allows it to translate `RelatedIssue` objects too

3. **SKILL.md** (OpenClaw bot instructions):
   - Updated API Actions Reference table to show `language_code?` on all data-fetching actions
   - Added critical instruction: "Pass `language_code` on every data-fetching API call"
   - Cleared OpenClaw sessions and restarted gateway for bot to pick up changes

## Tests

- 19 new tests total:
  - 11 regression tests for translated data in each affected action (Polish translations)
  - 8 structural guard tests (`it.each`) verifying all data-fetching actions accept `language_code` without validation error
- The structural guard test prevents this bug from recurring — if someone adds a new data-fetching action without `language_code`, the test fails
- Total: 1198 tests (up from 1155), all passing (~3.5s)

## Decisions

- **Accept-but-don't-translate pattern for actions/community:** `get_actions` and `get_community` accept `language_code` in their schema (for API consistency) but don't apply translations because actions and community data aren't translated in the DB yet. This means adding translations later requires zero API changes.
- **Structural guard over code review:** Instead of relying on code review to catch missing `language_code`, added a parametric test that explicitly enumerates all data-fetching actions and verifies they accept the param.
