# Session 54 — Server-Side Language Fallback for WhatsApp Bot

**Date:** 2026-02-27
**Branch:** `claude/server-side-language-fallback`
**PR:** [#129](https://github.com/Skye-Quiet-Riots/quiet-riots/pull/129)

## What was worked on

Fixed a regression where WhatsApp users writing in non-English languages (e.g., Spanish) received issue names in English. The root cause: 6 data-fetching bot actions defaulted to `'en'` when the OpenClaw LLM forgot to pass `language_code` — and the LLM doesn't always follow SKILL.md instructions reliably.

## Changes made

### `src/app/api/bot/route.ts`

1. **`resolveLocale()` helper** — resolves locale via priority chain: explicit `language_code` param > stored user preference (via phone lookup) > `'en'`. Logs when phone fallback is used.
2. **6 Zod schemas extended** with `phone: phoneField.optional()` — `search_issues`, `get_trending`, `get_issue`, `get_org_pivot`, `get_orgs`, `get_action_initiatives`
3. **6 action handlers updated** — replaced `(p.language_code as string) || 'en'` with `await resolveLocale(p)`
4. **`identify` auto-update** — when caller passes `language_code` for an existing user, the stored preference is now persisted (previously ignored for returning users)

### `src/app/api/bot/bot-api.test.ts`

16 new tests across 3 describe blocks:

- Phone fallback (8 tests): translations via phone, priority ordering, edge cases
- Identify auto-update (4 tests): updates, preserves, no-op for same value
- Regression (1 test): full identify→search flow reproducing the exact bug
- Structural guard (6 parametrized): all translating actions accept `phone`

### `~/.openclaw/skills/quiet-riots/SKILL.md`

- Added fallback instruction about including `phone` on data calls
- Updated API reference table with `phone?` on 6 actions + identify auto-update note

## Decisions made

1. **Per-action schemas, not top-level body field** — considered adding `phone`/`language_code` to `bodySchema` but rejected because it creates two places to pass the same thing and breaks existing SKILL.md curl examples.
2. **Keep `identify` auto-update** — security reviewer suggested not auto-updating to avoid overwriting explicit preferences. Kept it because: OpenClaw detects language from message text (not phone metadata), and the whole point is the LLM doesn't always call `set_language`.
3. **Don't modify `get_actions` or `get_community`** — they accept `language_code` but don't actually call translation functions. No point adding `phone` until they translate.
4. **`resolveLocale` as closure, not top-level** — placed inside the request handler like `resolveUser` so it captures `log` from scope.

## What was surprising

- Simon's user record had `language_code: "en"` even though he'd been testing in Spanish — the `identify` action was completely ignoring `language_code` for existing users (only used it for new user creation).
- The security review raised a valid concern about phone enumeration via language differences in API responses, but accepted it given the Bearer token auth gate.
- The original "fix" in session 34 was half-code (adding translation support — permanent) and half-prompt (telling the LLM to pass `language_code` — fragile). This session makes both halves deterministic.
