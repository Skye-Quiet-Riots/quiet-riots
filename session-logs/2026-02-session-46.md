## What Was Done

### 1. Phone OTP Delivery via WhatsApp Polling

**Problem:** Users couldn't sign in with their phone number on the web app because the OTP code never arrived. The Vercel `send-code` route tried to call a non-existent `send_whatsapp_message` bot action, which silently failed. The underlying issue is that Vercel serverless functions can't reach the local Mac's OpenClaw gateway to send WhatsApp messages.

**Solution:** Local polling architecture:
1. Vercel stores OTP codes in DB with `delivery_message` column (plaintext message for WhatsApp)
2. Local Mac polls bot API every 3s for undelivered codes
3. Sends via `openclaw message send --channel whatsapp`
4. Marks as delivered via `mark_code_delivered` bot action

**Files changed:**
- `migrations/021_phone_code_delivery.sql` — add `delivered_at` + `delivery_message` columns
- `src/lib/schema.ts` — add columns to CREATE TABLE
- `src/lib/queries/phone-verification.ts` — `getUndeliveredCodes()`, `markCodeDelivered()`, `clearDeliveryMessage()`, defence-in-depth NULLing
- `src/app/api/auth/phone/send-code/route.ts` — removed broken bot API fetch, stores delivery message
- `src/app/api/bot/route.ts` — added `get_undelivered_codes` + `mark_code_delivered` actions
- `scripts/deliver-otp-codes.sh` — polling script for local Mac

**Security model:**
- `delivery_message` NULLed after verification/expiry (defence in depth)
- `markCodeDelivered` uses atomic `WHERE delivered_at IS NULL` (race-safe)
- Bot API auth required for all delivery actions
- 5-min time-bounded exposure (codes expire)
- Prior codes invalidated when new code is requested (also NULLs their delivery messages)

## Decisions & Rationale

- **Polling over direct send:** Vercel serverless can't reach local Mac's OpenClaw gateway. Polling is simple, resilient, and uses existing bot API auth infrastructure.
- **3-second interval:** Balances responsiveness (user expects near-instant code) against API load (each poll is a single lightweight API call).
- **delivery_message in DB:** Storing plaintext is necessary for WhatsApp delivery. Mitigated by NULLing after use and time-bounded expiry.
- **Two-step code creation:** Route creates code first (getting the plaintext), then updates the record with the delivery message. This avoids restructuring `createVerificationCode` to accept a template.

## Gotchas Discovered

- **Bot test state leakage:** Bot API tests use `beforeAll` (shared DB), so phone verification codes created in one test persist to the next. Added `beforeEach` cleanup with `_resetVerificationCodes()` for the OTP delivery test group.
