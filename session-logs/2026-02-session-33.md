## What Was Done

### 1. Wired up database content translations (PR #74 — merged)

### 2. Debugged production translation issue

- After merging PR #74, translations weren't appearing on production
- Root cause: `npx vercel --prod` was run from the main repo checkout which was still at the pre-merge commit — it deployed stale code
- Fix: pulled `origin/main` in the main repo, then redeployed — translations appeared immediately ("Bagages perdus" in French instead of "Lost Luggage")

### 3. Debugged local file DB fallback

- During translation seeding, `npx tsx` silently fell back to `file:quiet-riots.db` because `.env.local` wasn't loaded
- Implemented two-layer prevention (PR #75 — merged):
  - `requireRemoteDb()` guard in `db-safety.ts` — blocks execution with clear error when DB URL is local
  - `scripts/with-staging-env.sh` — sources `.env.local` from main repo root (works from worktrees)
  - Wired guard into `seed-translations --apply`, `seed-assistants`, `seed-reference-data`
  - 4 new tests

## Decisions Made

- **`requireRemoteDb()` vs guard in every script:** Only wired into scripts that inherently need a remote DB with real data. `seed.ts` and `migrate.ts` are intentionally left without the guard — they legitimately run against local DBs during development.
- **Helper script approach:** `with-staging-env.sh` uses `git worktree list --porcelain` to find the main repo root, so it works from any worktree.

## Discoveries

- **`npx vercel --prod` deploys local filesystem, not git:** This is the #1 deployment footgun in the worktree workflow. The local checkout at `/Users/skye/Projects/quiet-riots` can be behind `origin/main` after PRs merge via GitHub API. Added to CLAUDE.md gotchas and auto memory.
- **Vercel auto-deploy sometimes doesn't trigger:** After merging PR #74, no production deployment was created for ~20 minutes. Manual `npx vercel --prod` was needed.
