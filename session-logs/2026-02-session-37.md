# Session 37 — Translated Synonym Search for All 44 Locales

**Date:** 2026-02-25
**Branch:** `claude/sharp-fermi` → PR #85 (merged)
**Tests:** 1519 (up from 1238)

## What Was Done

### Problem

Production had 0 synonym translations — non-English users searching in their language got no synonym matches. The `synonyms` table had 209 English terms but the translation system didn't cover them. Both the web app and WhatsApp bot were affected.

### Solution: Full synonym i18n pipeline

1. **CLAUDE.md protocols** — Added i18n Protocol (every new user-facing field must have translations from the outset) and Dual-Surface Protocol (every feature must work on both web and WhatsApp bot)

2. **Migration 016** — Added indexes on `synonyms(issue_id)` and `synonyms(term)` to prevent full table scans

3. **Extended seed-translations.ts** — Added `synonyms` support to `TranslationFile` type, `generateEnglishBaseline()`, and `applyTranslations()`. Synonyms stored as `entity_type='synonym'`, `entity_id=<synonym_id>`, `field='term'`

4. **Generated translations for all 44 locales** — Used 10 parallel sub-agents to translate ~209 English synonym terms per locale. All 45 translation files (including en.json baseline) now have a `synonyms` key

5. **Updated search query** — `buildTranslatedLikeClause()` now adds a translated synonym subquery for non-English searches. 6 args per word (vs 2 for English): issue name, English synonyms, translated issue names, translated synonym terms. This single change covers both web and bot since both call `getAllIssues()`

6. **Translated synonym display** — Added `translateSynonyms()` helper in `src/lib/queries/translate.ts`. Both `[locale]/issues/[id]/page.tsx` (web) and bot `get_issue` action now return translated synonym terms

7. **281 new tests** — Bot API tests (translated synonym search/display), issues API tests (locale-aware synonym search), seed-translations tests (synonym structure validation, security checks)

8. **Applied to staging + production** — Migration 016, 209 English synonyms (already existed), 18,612 translated synonym entries (423 per locale × 44 locales) on both environments

### Verification

- Web: `/api/issues?search=trenes+cancelados&locale=es` → returns Train Cancellations + Train Ticket Prices
- Bot: `search_issues` with `language_code=es`, query "trenes cancelados" → returns translated issue names
- Bot: `get_issue` with `language_code=es` → returns translated synonym terms (e.g., "tarifas ferroviarias")
- Health: `https://www.quietriots.com/api/health` → `{"status":"ok"}`

## Decisions & Reasoning

- **Synonym ID as entity_id:** Reused the existing translations table structure rather than creating a new table. Natural fit with the UNIQUE constraint `(entity_type, entity_id, field, language_code)`.
- **10 parallel sub-agents:** Translating 209 terms × 44 locales is ~9,000 translations. Parallelised across 10 agents to complete in minutes rather than an hour.
- **6 args per word for non-English:** The search query adds translated synonym matching via a subquery on the translations table, joining back to synonyms → issues. This avoids duplicating data while keeping the query performant.
- **Single query layer covers both surfaces:** Both web and bot call `getAllIssues()`, so the translated synonym search works on both without separate implementations.
