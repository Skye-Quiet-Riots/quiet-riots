## What Was Done

### Continued from Session 57

Session 57's PR #140 (386 new i18n tests + category_assistant DB translations) was merged and translations applied to both staging (24,035 translations) and production (29,425 translations).

### Problem Identified

User noticed English strings still appearing on the Spanish locale:
- **Issues page:** "16 AI & Human Assistant Pairs", "Every category has a dedicated AI agent and human organiser to help.", "Meet them →"
- **Assistants page:** English category badges (Transport, Health, Banking), English goal/description text on assistant cards

### Root Cause

Two separate translation gaps:
1. **Hardcoded English in 3 banner components** — `assistant-overview-banner`, `assistant-banner`, `assistant-detail-banner` had literal English strings instead of i18n keys
2. **Missing DB entity translation calls in 6 pages** — pages called `getAllAssistants()`/`getAssistantByCategory()` directly without calling `translateCategoryAssistants()`/`translateCategoryAssistant()` before rendering

WhatsApp bot and API routes were NOT affected — they correctly translate assistant data.

### Fixes Applied

1. **3 banner components** converted from sync to async server components with `getTranslations()` — 9 hardcoded English strings replaced with i18n keys
2. **6 page components** now call `translateCategoryAssistants`/`translateCategoryAssistant` for DB entity translation:
   - `issues/page.tsx`, `issues/[id]/page.tsx`
   - `organisations/page.tsx`, `organisations/[id]/page.tsx`
   - `assistants/page.tsx`, `assistants/[category]/page.tsx`
3. **assistant-card.tsx** now passes translated category label to `CategoryBadge`
4. **9 new `Assistants` namespace keys** added to `en.json` and translated to all 55 non-English locale files
5. **20 test cases** updated for async server component rendering pattern with `next-intl/server` mock
6. **"Zero Tolerance for Hardcoded English" rules** added to CLAUDE.md (5 mandatory rules)
7. **"Always merge and deploy" rule** added to CLAUDE.md PR lifecycle section

## Key Decisions

- **Banner components became async server components** — only way to call `getTranslations()` in a server component. Tests updated to use `const el = await Component({props}); render(el);` pattern.
- **Mock for `next-intl/server` in tests** — must include ALL namespaces used by ALL components in the test file, not just the ones being tested. Missing namespaces break unrelated tests.
- **Locale code mapping** — the actual locale file names differ from common assumptions (e.g., `zh-CN`/`zh-TW` not `zh`/`zh-Latn`, `tl` not `fil`, `ca`/`eu`/`gl`/`hr`/`sw` exist but `et`/`lt`/`lv`/`sr` do not).

## Test Count

2555 tests, 75 files — all passing.

## Gotchas Discovered

- **Locale file names don't match translation agent assumptions** — always check `ls messages/*.json` for actual codes before generating translations. The agent assumed 63 locales but only 55 non-English files exist, with different codes.
- **Mocking `next-intl/server` has global scope** — adding a mock for async banner components broke previously-passing tests for `HealthMeter`, `ActionInitiativeProgress`, and `TransactionList` because the mock didn't include their namespaces (`Health`, `ActionInitiativeProgress`, `Transactions`).
